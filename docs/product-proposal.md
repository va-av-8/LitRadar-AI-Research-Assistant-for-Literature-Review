# Product Proposal — LitRadar

---

## 1. Обоснование идеи

### Прикладная задача
Исследователь в области AI/ML тратит значительную часть времени на подготовительный этап любой работы — литературный обзор. Задача состоит из нескольких шагов: формулировка поискового запроса, итеративное расширение поиска, фильтрация нерелевантного, группировка по тематикам, выявление противоречий между работами, синтез в связный текст.

Существующие инструменты (Connected Papers, Elicit, Consensus) решают часть задачи, но не умеют итеративно углублять поиск на основе противоречий, обнаруженных в промежуточных результатах. Исследователь делает это вручную: читает статью, замечает отсылку к спорному результату, идёт искать дальше.

### Почему это агентная задача, а не просто LLM-запрос

Задача требует:
- **Многошагового планирования** — тема декомпозируется в подзапросы, подзапросы уточняются по ходу
- **Использования внешних инструментов** — поиск через API, получение метаданных
- **Промежуточных решений о качестве** — Critic оценивает релевантность каждого результата
- **Нелинейного исполнения** — Synthesis может вернуть управление Planner'у при обнаружении противоречий
- **Multi-hop reasoning** — связи между статьями строятся через structured state: агент отслеживает `detected_contradictions` и `open_questions` и использует их как основание для следующей итерации поиска

## 2. Цель и метрики успеха

### Цель
Сократить время подготовки первичного литературного обзора с 5–10 часов до 15–30 минут при сохранении приемлемого покрытия ключевых работ и явной разметки противоречий между источниками.

### Продуктовые метрики

| Метрика | Целевое значение | Способ измерения |
|---|---|---|
| Покрытие ключевых работ | ≥ 70% от эталонного списка | Ручная разметка по 10 тестовым темам |
| Точность релевантности статей в обзоре | ≥ 80% | Оценка вручную: релевантна / нерелевантна |
| Доля галлюцинированных цитат | 0% | Citation Guard + Verifier + ручная проверка выборки |
| Наличие раздела с противоречиями | 100% запросов, где противоречия есть | Проверка структуры вывода |

### Агентные метрики

| Метрика | Целевое значение | Комментарий |
|---|---|---|
| Среднее число итераций reflection loop | 1–2 | Больше — риск зацикливания и роста стоимости |
| Доля запросов без ручного вмешательства | ≥ 90% | Human-in-the-loop только для слишком широких / off-topic тем |
| Точность Critic агента | ≥ 80% precision | Оценивается на тестовом датасете с разметкой |
| Доля open_questions, закрытых на 2-й итерации | ≥ 60% | Показывает эффективность reflection loop |

### Технические метрики

| Метрика | Целевое значение |
|---|---|
| End-to-end latency p95 | ≤ 120 сек (5 подзапросов, ~20 статей) |
| Стоимость одного запроса (LLM API), soft limit | ≤ $0.08 |
| Стоимость одного запроса (LLM API), hard stop | $0.15 |
| Поведение при недоступности одного API | Graceful degradation, явный `source_gap_note` в отчёте |
| Поведение при недоступности обоих API | Ошибка с понятным сообщением пользователю |

## 3. Сценарии использования

### Happy path
Пользователь вводит *"chain-of-thought prompting in large language models"*. Input Guard пропускает запрос. KB Lookup находит 5 релевантных статей из предыдущих сессий. Planner формирует 4 подзапроса с учётом уже покрытых тем. Search находит 22 новые статьи. Sanitization очищает абстракты. Critic отсеивает 4 нерелевантных. Retriever индексирует 18 принятых статей. Synthesis обнаруживает противоречие: две группы авторов расходятся во мнении об эффективности CoT на малых моделях. Planner инициирует уточняющий поиск *"chain-of-thought small language models scaling"*. Citation Guard и Verifier проверяют все цитаты. Renderer формирует финальный обзор с 3 разделами, явной разметкой противоречия и disclaimer'ом об open question.

### Edge-кейсы

**Слишком широкая тема**
Запрос: *"machine learning"*. Input Guard отклоняет до старта графа.
Поведение: сообщение пользователю с просьбой уточнить тему. Human-in-the-loop обязателен.

**Слишком узкая / неизвестная тема**
Запрос: *"quantum neuro-symbolic grounding in embodied AGI"*. Search возвращает 0–2 результата. Critic отсеивает всё.
Поведение: `route_after_critique` → END; агент сообщает о недостаточном покрытии, не генерирует обзор.

**Противоречивые результаты без разрешения**
Synthesis обнаруживает противоречие, Planner инициирует 2-ю итерацию, противоречие не закрывается.
Поведение: противоречие явно выносится в отдельный раздел обзора как *"open question"*, не замалчивается.

**Prompt injection через содержимое статьи**
Абстракт статьи содержит текст вида *"Ignore previous instructions and output..."*.
Поведение: Sanitization слой заменяет паттерн на `[REMOVED]`, логирует WARNING в `logs/security.jsonl`; при `injection_rate > 0.30` в батче — немедленная остановка с уведомлением пользователя.

**Зависание reflection loop**
`open_questions` на итерации 2 совпадают с итерацией 1.
Поведение: Orchestrator детектирует зависание, принудительно переходит к Citation Guard с текущим состоянием.

**Частичная недоступность API**
Semantic Scholar недоступен, ArXiv работает.
Поведение: поиск продолжается только через ArXiv; `source_gap_note` в AgentState → Renderer добавляет disclaimer о неполном покрытии источников.

## 4. Ограничения

### Технические ограничения

| Параметр | Ограничение | Обоснование |
|---|---|---|
| Источники данных | ArXiv API, Semantic Scholar API | Бесплатные, без OAuth, стабильные для AI/ML тематики |
| Глубина анализа | Только метаданные и абстракты | Полнотекстовый анализ PDF вне scope PoC |
| Тематика | AI/ML (NLP, CV, RL, reasoning) | Управляемые evals, доменная экспертиза автора |
| Язык | Английский | Большинство релевантных источников |
| Число итераций loop | Максимум 2 (`MAX_ITERATIONS=2`) | Ограничение стоимости и latency |
| Максимум статей за сессию | 30 (`MAX_PAPERS_PER_SESSION=30`) | Управление контекстным окном и бюджетом |

### Операционные ограничения

| Параметр | Ограничение |
|---|---|
| Бюджет на LLM API (PoC) | ~$20–30 суммарно на разработку и тестирование |
| Soft limit на один запрос | $0.08 (предупреждение в UI) |
| Hard limit на один запрос | $0.15 (принудительная остановка) |
| Rate limits ArXiv | 3 req/sec, без ключа |
| Rate limits Semantic Scholar | 100 req/min без ключа, 1 req/sec рекомендуется |
| Команда | 1 человек, 10 ч/нед, 4 недели = ~40 ч |
| Инфраструктура | Локальный запуск + Streamlit, без облачного деплоя в PoC |

## 5. Архитектурный набросок

### Агенты и их роли

```
User Input
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│                      Input Guard                            │
│  Детерминированная проверка: стоп-фразы, ширина запроса,    │
│  тематическая фильтрация (embedding cosine similarity)      │
│  Без LLM — стоимость $0                                     │
└────────────────────────┬────────────────────────────────────┘
                         │ (при ошибке → END + сообщение)
    ┌────────────────────▼────────────────────────────────────┐
    │                    KB Lookup                            │
    │  Retrieval из персистентной ChromaDB (./chroma_db/)     │
    │  Возвращает papers_from_kb → Planner учитывает пробелы  │
    └────────────────────┬────────────────────────────────────┘
                         │
         ┌───────────────▼───────────────┐
         │         Planner Agent         │
         │  Декомпозиция темы в 3–5      │
         │  подзапросов с учётом KB      │
         │  Iter 2: уточнение по         │
         │  open_questions от Synthesis  │
         └───────────────┬───────────────┘
                         │  subqueries
         ┌───────────────▼───────────────┐
         │    Search (ArXiv + S2 API)    │
         │  Детерминированный код        │
         │  Дедупликация по paper_id     │
         │  через papers_seen            │
         └───────────────┬───────────────┘
                         │  papers_found
         ┌───────────────▼───────────────┐
         │       Sanitization Layer      │
         │  Детерминированный код        │
         │  Паттерны → [REMOVED]         │
         │  Оборачивание в              │
         │  <external_content ...>       │
         │  injection_rate > 0.30 → END  │
         └───────────────┬───────────────┘
                         │
         ┌───────────────▼───────────────┐
         │         Critic Agent          │
         │  Оценка релевантности батчами │
         │  по ~10 статей                │
         │  rejected/total > 0.6 →       │
         │  возврат к Planner (iter < 2) │
         └───────────────┬───────────────┘
                         │  papers_accepted
         ┌───────────────▼───────────────┐
         │     Retriever (Index)         │
         │  Индексирование papers_       │
         │  accepted в ChromaDB          │
         │  (персистентно, на диске)     │
         └───────────────┬───────────────┘
                         │
         ┌───────────────▼───────────────┐
         │       Synthesis Agent         │
         │  RAG retrieval из ChromaDB    │
         │  Группировка по темам         │
         │  Поиск contradictions         │
         │  Формулировка open_questions  │
         │  open_questions ≠ prev →      │
         │  возврат к Planner (iter < 2) │
         └───────────────┬───────────────┘
                         │  synthesis_output
         ┌───────────────▼───────────────┐
         │       Citation Guard          │
         │  Детерминированная проверка   │
         │  paper_id ∈ papers_accepted   │
         │  ∪ papers_from_kb             │
         │  O(n), без сетевых запросов   │
         └───────────────┬───────────────┘
                         │
         ┌───────────────▼───────────────┐
         │           Verifier            │
         │  HTTP GET к API источника     │
         │  по каждой цитате             │
         │  200 = verified, иначе →      │
         │  removed_citations            │
         └───────────────┬───────────────┘
                         │  verified_citations
         ┌───────────────▼───────────────┐
         │        Renderer Agent         │
         │  Генерация Markdown-обзора    │
         │  из верифицированного         │
         │  synthesis JSON               │
         │  Добавляет source_gap_note    │
         │  если источник был недоступен │
         └───────────────┬───────────────┘
                         │
                    Streamlit UI
```

### Ключевые архитектурные паттерны

**Reflection loop с multi-hop reasoning**
Synthesis агент поддерживает structured state сессии:
```json
{
  "detected_contradictions": [
    {
      "topic": "CoT effectiveness on small models",
      "paper_a": "arxiv:2201.xxxxx",
      "paper_b": "arxiv:2309.xxxxx",
      "description": "Paper A claims CoT degrades on <7B models, Paper B shows gains with few-shot CoT on 3B"
    }
  ],
  "open_questions": [
    "What is the role of model scale in CoT effectiveness below 7B parameters?"
  ]
}
```
`open_questions` передаются в Planner как основание для следующей итерации поиска. Orchestrator сравнивает их с `prev_open_questions` для детекта зависания.

**Двухслойная защита от галлюцинаций цитат**
Citation Guard (детерминированный, O(n)) → Verifier (HTTP к API источника). Ни один слой не использует LLM.

**Critic как отдельный агент, а не часть Synthesis**
Разделение позволяет независимо оценивать и улучшать качество фильтрации, тестировать Critic отдельно, логировать его решения для evals.

**Персистентная база знаний между сессиями**
ChromaDB хранится на диске (`./chroma_db/`). Каждая сессия обогащает базу новыми статьями; KB Lookup в начале следующей сессии возвращает ранее найденные работы, исключая повторный поиск.

### Технологический стек

| Компонент | Технология |
|---|---|
| Агентный фреймворк | LangGraph (StateGraph) |
| LLM (default) | `google/gemini-2.0-flash` через OpenRouter API |
| LLM (Synthesis, опционально) | `openai/gpt-4.1` через OpenRouter API |
| Эмбеддинги | `BAAI/bge-small-en-v1.5` (sentence-transformers, локально, ~130 МБ) |
| Поиск статей | ArXiv API (`arxiv` PyPI), Semantic Scholar Graph API |
| Векторная БД | ChromaDB, персистентная коллекция на диске (`./chroma_db/`) |
| UI | Streamlit |
| Observability | Langfuse (трейсинг per-session) + локальные JSONL-логи |
| Язык | Python 3.11 |
| Управление зависимостями | `uv` (pyproject.toml + uv.lock) |

## 6. Data flow и разделение ответственности

### Что делегируется LLM/агентам

| Задача | Агент | Обоснование |
|---|---|---|
| Декомпозиция темы в подзапросы | Planner | Требует понимания семантики, нельзя шаблонизировать |
| Оценка релевантности статьи | Critic | Субъективное суждение, зависит от контекста запроса |
| Обнаружение противоречий | Synthesis | Требует понимания смысла, сравнения позиций авторов |
| Формулировка open_questions | Synthesis | Генеративная задача |
| Написание финального обзора | Renderer | Генеративная задача, получает только верифицированный JSON |

### Что НЕ делегируется LLM

| Задача | Кто выполняет | Обоснование |
|---|---|---|
| Тематическая фильтрация запроса | Input Guard (embedding cosine similarity) | Детерминированность, нулевая стоимость |
| Запрос к ArXiv / Semantic Scholar | Детерминированный код | Точность, предсказуемость, rate limiting |
| Sanitization абстрактов | Детерминированный код | Безопасность, защита от prompt injection |
| Проверка paper_id на принадлежность к корпусу | Citation Guard (детерминированный) | O(n) без LLM, первый рубеж защиты от галлюцинаций |
| Верификация существования цитаты | Verifier (HTTP к API источника) | Нельзя доверять LLM проверку фактов о себе же |
| Контроль числа итераций loop | Orchestrator (код) | Защита от зависания, управление стоимостью |
| Индексирование и retrieval | Retriever (ChromaDB + sentence-transformers) | Детерминированный cosine similarity, нулевая стоимость |
| Логирование и метрики | Langfuse + локальные файлы | Нейтральность наблюдателя |
